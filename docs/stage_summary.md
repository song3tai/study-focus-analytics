# Study Focus Analytics 当前阶段总结

## 当前阶段判断

当前这一版已经进入“基础功能初步稳定、可以持续打磨产品体验”的阶段。

从现状来看，项目已经不再只是检测能力原型，而是已经具备一条可运行、可展示、可联调的完整分析链路：

`视频输入 -> 人体检测 -> 场景特征提取 -> 状态判断 -> 专注度估计 -> 统计汇总 -> Web 展示`

结合当前代码、接口、Dashboard 和测试情况，可以把这一阶段定义为：

- V1 主链路已打通
- 视频处理与播放基本稳定
- Web UI 已可用
- 已具备继续做结果表达、体验优化和产品化推进的条件

## 当前已具备的基本功能

### 1. 多输入源接入

当前系统已经支持以下输入方式：

- 本地视频文件输入
- 摄像头输入
- RTSP 实时流输入

这些输入已经统一接入到同一条分析链路中，便于后续复用相同的数据模型和行为逻辑。

### 2. 视频检测与分析双模式

当前 CLI 已支持两种基础运行模式：

- `detect`：检测预览模式，重点查看检测结果
- `analyze`：完整分析模式，执行检测、状态分析、专注度估计和统计汇总

这说明系统已经不只是“能跑检测”，而是具备了正式分析模式。

### 3. 基于 YOLO 的单人检测能力

当前项目已经完成基于 YOLO 的 person 检测能力接入，并输出结构化检测结果，用于后续行为分析，而不再停留在单纯画框展示层。

### 4. 场景特征提取与状态判断

当前项目已经具备基础行为分析能力，可以围绕单人学习 / 工作场景输出以下状态：

- `unknown`
- `present`
- `away`
- `studying`

这意味着系统已经能从“看到人”进一步推进到“理解当前处于什么状态”。

### 5. 专注度估计

当前系统已经具备基础专注度估计能力，可输出：

- `focus_score`
- `focus_level`
- `reasons`

这一能力已经可以支持前端展示当前专注状态，并为后续结果解释提供基础。

### 6. 时长统计与汇总

当前项目已经可以对分析过程做基础统计汇总，包括：

- 总分析时长
- 在岗时长
- 离岗时长
- `studying` 时长
- 离岗次数
- 平均 / 最大 / 最小专注度

这说明系统已经能给出用户真正关心的阶段性结果，而不只是逐帧判断。

### 7. Web 服务与可视化界面

当前系统已经具备本地 Web 展示能力，包含：

- FastAPI REST API
- WebSocket 实时状态推送
- MJPEG 预览流
- Dashboard 页面

当前 Dashboard 已经可以展示：

- 服务状态与会话状态
- 当前分析状态
- 专注度与等级
- 主要统计结果
- 一部分解释性与调试信息

这意味着项目已经具备完整的前后端联调能力和可演示能力。

### 8. 测试与稳定性基础

当前仓库已经具备较完整的测试覆盖，核心模块、Web API 和分析主链路已有自动化验证。

在当前阶段，本地测试结果为：

- `30 passed`

这说明当前版本已经具备继续迭代的工程基础。

## 当前阶段的整体评价

如果用一句话概括当前状态，可以表述为：

当前版本已经完成了 V1 的核心分析闭环，视频处理、播放、分析和展示链路都已经初步稳定，项目已经从“功能原型”进入“结果打磨和产品化推进”阶段。

现阶段最重要的工作不再是继续快速堆更多能力，而是围绕以下方向把已有能力做扎实：

- 让结果更稳定
- 让结果更容易理解
- 让离线分析和实时分析都更贴近真实使用场景
- 让前端呈现更完整地表达分析价值

## 下一步计划

### 1. 增加视频文件的“快速分析模式”

这是下一步非常值得加入的功能。

当前视频文件分析更接近“边播放边分析”的实时体验。后续建议为视频文件输入增加两种可选模式：

- 正常分析模式：按当前方式播放并同步分析，适合演示和观察过程
- 快速分析模式：不按实时播放速度展示，而是尽可能以最快速度跑完整个视频，并在完成后立即给出分析结果

这个功能的价值很明确：

- 更适合离线复盘场景
- 更适合长视频批量分析
- 用户无需等待整段视频按原速播放完成
- 能把“演示模式”和“结果模式”明确区分开

后续这个功能建议优先只针对 `video_file` 输入开放，而不作用于摄像头和 RTSP 实时流。

### 2. 优化离线分析结果输出

当快速分析模式加入后，离线分析结果页就会变得更重要。下一步建议加强：

- 分析完成后的汇总页
- 时间线展示
- 事件列表展示
- 更清晰的阶段统计卡片
- 对专注度和状态变化的解释

这样用户不一定要盯着实时画面，也能直接消费最终结果。

### 3. 优化状态机与专注度规则稳定性

当前主链路已经可用，下一步应该继续优化：

- `present / away / studying` 切换稳定性
- 去抖逻辑
- ROI 相关判断
- 专注度分数和等级的合理性
- 原因说明的可解释性

这会直接决定最终结果是否可信、是否好用。

### 4. 完善事件流与历史查询能力

为了支持更完整的结果展示，后续建议逐步补齐：

- 事件流输出
- 时间线接口
- 历史分析结果查询
- 离线分析记录管理

这部分能力会让系统从“实时仪表盘”继续走向“可复盘工具”。

### 5. 持续优化 Dashboard 表达能力

当前 UI 已经出来了，下一阶段建议重点做信息表达优化，而不是推翻重做。可以继续增强：

- 当前状态展示
- 分析结果摘要
- 错误和异常提示
- 解释性文案
- 离线分析完成后的结果承接

目标是让界面不仅“能看”，还要“看得懂、看得出价值”。

### 6. 推进本地产品化封装

当主链路和离线分析体验进一步稳定后，再继续推进：

- Electron 桌面封装
- 更完整的本地运行体验
- 配置、结果查看和历史管理的一体化

这一步更偏产品化，不需要早于分析链路本身的稳定化。

## 建议的下一阶段目标表述

如果要把下一阶段目标写成一句更适合项目内部沟通的话，可以表述为：

下一阶段将在保持当前主链路稳定的基础上，重点补强离线分析体验，增加视频文件快速分析模式，并继续提升状态判断、结果汇总和 Dashboard 展示的完整度。

## 一句话总结

当前版本已经具备“可稳定演示的分析 MVP”，下一步重点不是继续扩边界，而是围绕“快速离线分析、结果表达、稳定性优化、产品化体验”把现有能力做深做实。
